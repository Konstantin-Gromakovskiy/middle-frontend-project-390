# Repository Guidelines

## Структура проекта

Основное Next.js-приложение находится в `code/`. Исходный код App Router расположен в `code/src/app/`: страницы и layout -- в `.tsx`, локальные стили -- в `.module.css`, общие -- в `globals.css`. Статические файлы храните в `code/public/`. Unit-тесты находятся в `code/tests/unit/`, браузерные сценарии Playwright -- в `code/tests/e2e/`. API-контракт для мок-сервера лежит в `contract/openapi.yaml`.

В корне находятся `Makefile`, CI-конфигурации и зависимости мок-сервера. Не изменяйте `.github/workflows/hexlet-check.yml`: этот файл создается и используется Hexlet.

## Команды разработки

Установите зависимости из корня: `make install`. Основные команды также запускаются из корня:

- `make dev` -- запускает Next.js в режиме разработки.
- `make mock-server` -- поднимает Prism по `contract/openapi.yaml`.
- `make lint` -- проверяет фронтенд ESLint.
- `make build` -- собирает production-версию приложения.
- `make test` -- последовательно запускает Vitest и Playwright.

Для точечного запуска используйте `cd code && npm run test:unit` или `cd code && npm run test:e2e`. Перед E2E-тестами приложение должно быть доступно по `APP_URL` (в CI: `http://127.0.0.1:3000`).

## Стиль кода

Пишите TypeScript в строгом режиме. Следуйте существующему стилю: 2 пробела, двойные кавычки, точка с запятой. Компоненты и файлы компонентов называйте в `PascalCase`, хуки -- с префиксом `use`, тесты -- `*.test.ts` и `*.spec.ts`. Используйте импорт `@/*` для модулей из `code/src/`. Перед коммитом запускайте `make lint`; правила Next.js и TypeScript определены в `code/eslint.config.mjs`.

## Тестирование

Добавляйте unit-тесты на поведение функций и компонентов рядом по смыслу в `code/tests/unit/`. E2E-сценарии размещайте в `code/tests/e2e/`, проверяя пользовательский результат через доступные роли и текст, а не внутренние CSS-селекторы. Новая функциональность должна сопровождаться релевантным тестом.

## Коммиты и pull request

История использует короткие описательные сообщения на русском, например: `добавил тесты, настроил ci` или `Поправил отступы в ci.yml`. Описывайте одно логическое изменение в коммите. В PR укажите цель, выполненные проверки и связанную задачу; для изменений интерфейса приложите скриншоты. Не включайте в PR артефакты `.next/`, `playwright-report/` и `test-results/`.
